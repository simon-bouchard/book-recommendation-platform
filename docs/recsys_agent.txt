# Recommendation System Refactoring Specification

## Goal
Improve LLM decision-making accuracy by splitting complex agents into smaller, focused components with clearer responsibilities.

## Three Main Changes

### Change 1: Split Retrieval Agent (Query Analyzer + Tool Executor)
### Change 2: Split Curation Agent (Ranking + Prose Generation)  
### Change 3: Split subject_hybrid_pool Tool (Popular Books + Subject Search)

---

# CHANGE 1: Split Retrieval Agent

## Problem
Current `RetrievalAgent` has too many responsibilities:
- Understand query intent
- Map query type → tool choice
- Execute tools with good parameters
- Evaluate results
- Decide when to stop

This causes mistakes like using `book_semantic_search` for vague queries ("recommend something") when it should use `als_recs` or `subject_hybrid_pool`.

## Solution Architecture

```
User Query
    ↓
QueryAnalyzer (new component)
  - Single LLM call
  - Classifies query type
  - Recommends 1-2 tools
  - Fast, deterministic
    ↓ outputs strategy dict
ToolExecutorAgent (modified RetrievalAgent)
  - Receives strategy as input
  - Executes recommended tools
  - Uses ReAct loop
  - Can adjust if results poor
    ↓ outputs candidates
CurationAgent (existing)
```

## Files to Create

### 1. `app/agents/infrastructure/recsys/query_analyzer.py`

**Purpose:** Lightweight query classifier that runs before tool execution.

**Class:** `QueryAnalyzer`

**Key Method:**
```
analyze(query: str, available_tools: list[str]) -> dict
```

**Returns:**
```python
{
    "query_type": "vague" | "descriptive" | "genre",
    "recommended_tools": ["tool_name", ...],
    "confidence": "high" | "medium" | "low", 
    "reasoning": "brief explanation"
}
```

**Query Type Definitions:**
- **vague**: Generic requests, no specific vibes/themes (e.g., "recommend something", "what to read")
- **descriptive**: Mentions atmosphere, mood, vibe (e.g., "cozy fantasy", "dark mystery")
- **genre**: Mentions specific subjects/genres (e.g., "historical fiction", "sci-fi")

**Tool Recommendation Logic:**
- vague + als_recs available → ["als_recs"]
- vague + als_recs NOT available → ["popular_books"]  
- descriptive → ["book_semantic_search"]
- genre → ["subject_id_search", "subject_hybrid_pool"]

**Implementation Details:**
- Use `get_llm(tier="small")` for speed/cost
- Single LLM call with JSON mode
- ~200 token prompt
- Fallback strategy if JSON parsing fails
- Log analysis result to chatbot.log

**Prompt Structure:**
```
Analyze this book recommendation query and recommend retrieval strategy.

QUERY: "{query}"
AVAILABLE TOOLS: {available_tools}

[Query type definitions]

Return JSON:
{
  "query_type": "vague|descriptive|genre",
  "recommended_tools": ["tool"],
  "confidence": "high|medium|low",
  "reasoning": "explanation"
}
```

### 2. Modify `app/agents/infrastructure/recsys/retrieval_agent.py`

**Changes to `RetrievalAgent` class:**

**Add to `__init__`:**
- Create `self.query_analyzer = QueryAnalyzer()` instance

**Override `execute()` method:**
1. Call analyzer before starting ReAct loop:
   ```
   strategy = self.query_analyzer.analyze(
       query=request.user_text,
       available_tools=list(self.tool_executor._tools.keys())
   )
   ```

2. Store strategy in state:
   ```
   state.intermediate_outputs["strategy"] = strategy
   ```

3. Pass to existing graph execution (ReAct loop continues normally)

**Modify `_build_current_situation()`:**
- Add strategy information at the top of situation prompt
- Show: query_type, recommended_tools, reasoning
- Add instruction: "You should generally follow this strategy unless results are poor"

**New System Prompt:**
Replace complex decision tree with:
```
You are the Tool Executor for book retrieval.

You receive a recommended strategy from the Query Analyzer and execute it.

Your job:
1. Execute the recommended tool(s)
2. Check if you have 60-120 candidates
3. Decide if additional tools needed (only if results poor)
4. Stop when sufficient candidates gathered

FOLLOW the recommended strategy unless:
- Tool returns very few results (< 20 books)
- Results seem off-topic or low quality
- You have strong reason to try different approach

Target: 60-120 candidates for curation stage.
```

**Expected Behavior:**
- Analyzer says "vague → use als_recs"
- Executor sees this, executes `als_recs(top_k=120)`, stops
- Executor has clear guidance but can override with reasoning

# CHANGE 2: Split Curation Agent

## Problem
Current `CurationAgent` does two distinct tasks in one LLM call:
1. Filter/rank candidates (complex scoring logic)
2. Write prose descriptions (creative writing)

This makes the prompt very long (~600 tokens) and mixes analytical + creative tasks.

## Solution Architecture

```
Candidates from Retrieval
    ↓
RankingAgent (new)
  - Filter poor quality books
  - Score by relevance
  - Order by score
  - Select top 6-30 books
  - Returns ordered IDs only
    ↓ ordered book IDs
ProseAgent (new)
  - Receives top 8-12 books
  - Writes descriptions
  - Returns formatted text
    ↓ final response
```

## Files to Create

### 3. `app/agents/infrastructure/recsys/ranking_agent.py`

**Purpose:** Filter and rank candidates analytically.

**Class:** `RankingAgent`

**Key Method:**
```
execute(
    request: AgentRequest,
    candidates: list[BookRecommendation],
    retrieval_summary: list[dict]
) -> list[int]  # Returns ordered book IDs only
```

**Responsibilities:**
- Filter out non-English, corrupted, or noise books
- Score by relevance (subjects, tones, themes)
- Diversify (avoid too many same author)
- Return 6-30 book IDs in ranked order

**Prompt Structure (~300 tokens):**
```
You rank book candidates by relevance.

USER QUERY: {query}
RETRIEVAL HISTORY: {summary}
CANDIDATES: {candidates_json}

Tasks:
1. Filter poor quality (non-English, corrupted, missing data)
2. Score by relevance to query
3. Diversify authors (max 3 per author unless requested)
4. Return 6-30 top books

Scoring weights:
- Explicit constraints (subjects/tones/authors): 40%
- Query theme alignment: 35%
- Metadata completeness: 10%
- Diversity: 15%

Return JSON:
{
  "book_ids": [1234, 5678, ...],
  "reasoning": "brief explanation"
}
```

**Implementation:**
- Use `get_llm(tier="large")` for complex reasoning
- JSON mode
- Return just IDs, NO prose

### 4. `app/agents/infrastructure/recsys/prose_agent.py`

**Purpose:** Write descriptions for top-ranked books.

**Class:** `ProseAgent`

**Key Method:**
```
execute(
    request: AgentRequest,
    ordered_books: list[BookRecommendation]  # Already ranked, top 8-12
) -> str  # Returns formatted prose
```

**Responsibilities:**
- Describe top 8-12 books only
- Explain WHY each fits the query
- Format with markdown (bold titles, line breaks)
- Warm, knowledgeable tone

**Prompt Structure (~250 tokens):**
```
Write descriptions for these top book recommendations.

USER QUERY: {query}
BOOKS (already ranked): {books_json}

Format:
[Optional intro sentence]

**Book Title** by Author - Why it fits (1-2 sentences).

**Book Title** by Author - Why it fits (1-2 sentences).

[Optional conclusion]

Rules:
- Bold titles with **Title**
- Line break between each book
- Explain relevance to query
- Warm, knowledgeable tone
- NO technical details (item_idx, etc.)
```

**Implementation:**
- Use `get_llm(tier="medium")` - simpler task, can use smaller model
- Takes top 8-12 from ranked list
- Returns plain text (not JSON)

### 5. Modify `app/agents/infrastructure/recsys/orchestrator.py`

**Changes to `RecommendationAgent`:**

**Add to `__init__`:**
```python
self.ranking_agent = RankingAgent()
self.prose_agent = ProseAgent()
```

**Modify `execute()` method:**

**Replace Stage 2 (current curation) with:**
```
# Stage 2a: Ranking
ordered_ids = self.ranking_agent.execute(
    request=request,
    candidates=candidates,
    retrieval_summary=retrieval_summary
)

# Reorder candidates to match ranking
ordered_books = [
    book for book_id in ordered_ids 
    for book in candidates if book.item_idx == book_id
]

# Stage 2b: Prose
prose_text = self.prose_agent.execute(
    request=request,
    ordered_books=ordered_books[:12]  # Top 12 only
)

# Build final response
return AgentResponse(
    text=prose_text,
    target_category="recsys",
    book_recommendations=ordered_books,  # All ranked books
    success=True,
    policy_version="recsys.orchestrator"
)
```

**Benefits:**
- Ranking uses large model, prose uses medium (cost savings)
- Can retry prose without re-ranking
- Each prompt 50% shorter
- Clearer separation of analytical vs creative tasks

# CHANGE 3: Split subject_hybrid_pool Tool

## Problem
`subject_hybrid_pool` does TWO different things based on optional parameter:
- With `fav_subjects_idxs` → subject-filtered books
- Without `fav_subjects_idxs` → popular books (Bayesian ranking)

This confuses the LLM. Same tool name, completely different behaviors.

## Solution
Create two separate tools with single, clear purposes.

## Files to Modify/Create

### 6. Create `app/agents/tools/popular_books.py`

**Purpose:** Return top books by Bayesian average rating.

**Tool Definition:**

**Name:** `popular_books`

**Description:** 
```
Returns most popular books ranked by Bayesian average rating.
Use for cold users or vague queries when personalization not available.
```

**Parameters:**
```python
def execute(self, top_k: int = 100) -> dict:
    """
    Args:
        top_k: Number of books to return (default 100)
    
    Returns:
        {
            "books": [
                {
                    "item_idx": 1234,
                    "title": "...",
                    "author": "...",
                    "year": "...",
                    "score": 0.95,
                    ...
                }
            ]
        }
    """
```

**Implementation:**
- Call existing Bayesian ranking code
- Same backend as current `subject_hybrid_pool(fav_subjects_idxs=None)`
- Returns books with full metadata

### 7. Modify `app/agents/tools/subject_tools.py`

**Changes to `subject_hybrid_pool`:**

**Make `fav_subjects_idxs` REQUIRED:**
```python
def execute(
    self,
    fav_subjects_idxs: list[int],  # No longer Optional
    top_k: int = 100,
    weight: float = 0.7
) -> dict:
```

**Update Description:**
```
Returns books matching specific subjects/genres, weighted by popularity.
REQUIRES subject IDs - use subject_id_search first to get IDs.
```

**Remove the "no subjects" fallback:**
- If called without subjects, raise clear error
- Forces LLM to use `popular_books` instead

### 8. Update Tool Registry

**In `app/agents/tools/registry.py`:**

**Add to internal tools registration:**
```python
if internal:
    from app.agents.tools.popular_books import PopularBooksRetrieval
    tools.append(PopularBooksRetrieval().to_tool_definition())
```

**Ensure availability:**
- `popular_books` available to ALL users (warm + cold)
- `als_recs` only available to warm users (10+ ratings)
- `subject_hybrid_pool` requires subject IDs always

## Update `recsys.retrieval.md`

**Replace "Available Tools" section:**

OLD:
```
**subject_hybrid_pool** - Genre/subject-based or popular books

- Without subject IDs: popular books (cold user fallback)
```

NEW:
```
**popular_books** - Most popular books by Bayesian rating

- Arguments: top_k (default 100)

- REQUIRES subject IDs (use subject_id_search first)
- Arguments: fav_subjects_idxs (list[int]), top_k, weight
```

**Update strategy examples:**

Vague query + cold user:
```json
{
  "action": "tool_call",
  "tool": "popular_books",
```
