### Producer Features

**Implemented:**
- Circuit breaker (60s timeout, 3 failures to open)
- Dual-write mode (env: `ENRICH_DUAL_WRITE_JSONL`, `ENRICH_CUTOVER_COMPLETE`)
- Fail-fast post-cutover
- Retry with exponential backoff
- 12 structured error codes
- Automatic metadata injection

**Cutover Flow:**
1. Dual-write: Kafka + JSONL (validation)
2. Monitor parity, lag, metrics
3. Switch to Kafka-only
4. Deprecate JSONL after 1-2 weeks

### Streaming Consumer

**Processing Pattern:**
- Micro-batches every 30-60 seconds
- Consumer groups: `cg.enrichment.sql.v1`, `cg.enrichment.errors.v1`
- Backpressure: `maxOffsetsPerTrigger` (1000 results, 500 errors)

**Critical Implementation: Two-Phase Commits**

This prevents orphaned records and was NOT in original plan:

**Phase 1: Dictionary Tables**
1. Upsert `llm_subjects` (INSERT IGNORE)
2. Upsert `vibes` (INSERT IGNORE)
3. **COMMIT** ← Critical checkpoint

**Phase 2: Link Tables**
4. Upsert `book_llm_subjects` (JOIN to resolve subject → idx)
5. Upsert `book_vibes` (JOIN to resolve text → vibe_id)
6. Upsert `book_tones`, `book_genres`
7. **COMMIT**

**Other Features:**
- Chunked inserts (1000 rows/batch) to avoid max_allowed_packet
- Idempotent upserts (INSERT ... ON DUPLICATE KEY UPDATE)
- Parameterized queries for safety

### SQL Schema

**Error Tracking Table:**
```sql
CREATE TABLE enrichment_errors (
    item_idx INT PRIMARY KEY,
    first_seen_at DATETIME NOT NULL,
    last_seen_at DATETIME NOT NULL,
    occurrence_count INT DEFAULT 1,

    error_code VARCHAR(32) NOT NULL,

```

**run_history format:** `[{run_id, timestamp, error_code}, ...]`

### Error Taxonomy (12 Codes)

- `INVALID_GENRE` - Not in ontology
- `INVALID_TONE_ID` - Not in ontology
- `VIBE_TOO_LONG` / `VIBE_TOO_SHORT` - Word count violations
- `SUBJECT_COUNT_WRONG` - Count violation
- `TONE_COUNT_WRONG` - Count violation
- `JSON_PARSE` - Unparseable response
- `HTTP_401` - Auth failure
- `RATE_LIMIT` - API limit
- `TIMEOUT` - API timeout
- `DB_TIMEOUT` - Database timeout
- `UNKNOWN` - Catch-all

## Phase 2: Intelligent Quality Tiering ✅ COMPLETED

### Five-Tier System (AS IMPLEMENTED)

| Tier | Score | Description | Expected % |
|------|-------|-------------|-----------|
| **RICH** | 60-100 | Long desc + many OL subjects | 60-70% |
| **SPARSE** | 30-59 | Medium metadata | 20-25% |
| **MINIMAL** | 10-29 | Weak metadata | 5-10% |
| **BASIC** | 1-9 | Very sparse (NEW tier) | 3-5% |
| **INSUFFICIENT** | 0 | Missing title/author | <1% |

**Note:** BASIC tier added during implementation (not in original 4-tier plan)

### Scoring Algorithm

**Description Points (0-60):**
- ≥100 words → 60 pts
- 50-99 words → 40 pts
- 20-49 words → 20 pts
- 10-19 words → 10 pts
- <10 words → 0 pts

**OL Subjects Points (0-40):**
- ≥10 subjects → 40 pts
- 7-9 subjects → 30 pts
- 5-6 subjects → 20 pts
- 3-4 subjects → 10 pts
- <3 subjects → 0 pts

**Total: desc_points + ol_points (0-100)**

### Quality Detection

- Placeholder text filtering (regex: "N/A", "No description", etc.)
- Encoding corruption (replacement chars, >30% special chars, all-caps)
- OL subject deduplication

### Tier Requirements

**RICH (60-100):**
- Subjects: 5-12 required
- Tones: 2-4 required
- Vibe: 6-20 words required (validator range; prompts target 8-12 for quality)
- Genre: required
- Validation: strict

**SPARSE (30-59):**
- Subjects: 3-6 required
- Tones: 0-2 optional
- Vibe: 3-10 words optional (validator range; prompts target 4-8 for quality)
- Genre: required
- Validation: lenient on optional

**MINIMAL (10-29):**
- Subjects: 1-3 required
- Tones: 0-1 optional
- Vibe: MUST be empty
- Genre: required
- Uses LLM with restrictive prompt (NOT rule-based as originally planned)

**BASIC (<10):**
- Subjects: 0-1 optional
- Tones: MUST be empty
- Vibe: MUST be empty
- Genre: required
- Goal: genre only

**Note on Vibe Ranges:** Validators use wider ranges (6-20 for RICH) while prompts instruct narrower targets (8-12 for RICH). This gives wiggle room while encouraging quality.

### Ontology

**Tones v2 (36 options, 6 buckets) - PRIMARY:**
```
PACING:        slow-burn, fast-paced, episodic, meandering
MOOD:          dark, lighthearted, melancholic, hopeful, tense, nostalgic
INTENSITY:     intense, cozy, gritty, gentle
HUMOR:         witty, satirical, absurd, dry
STYLE:         lyrical, conversational, experimental, descriptive, minimalist
ACCESSIBILITY: accessible, challenging, dense
```

**Tones v1 (55 options, flat list) - LEGACY:**
- Backward compatible via `ENRICHMENT_ONTOLOGY_VERSION=v1`

**Genres (39 options):**
- Single genre per book
- Prompt includes mappings:
  - astrology/tarot → religion-spirituality
  - astronomy → science-nature
  - self-improvement → psychology-self-help
  - sports/fitness → lifestyle

### Intelligent Retry System (NEW)

**Major enhancement not in original plan**

**Process:**
1. First attempt: standard prompt
2. Validation failure: extract error details
3. Retry: feedback prompt with original response + requirements
4. Track with `retry_count` in metadata
5. Both fail: DLQ with full context

**Retryable Errors:**
- Vibe length violations
- Subject/tone count violations
- Invalid genre

**Impact:**
- 30-40% reduction in DLQ volume
- >80% retry success rate
- Significantly improved quality

### Prompt Structure

**System Prompt:**
- Return strict JSON only
- All outputs in English
- Don't invent facts
- Catalog subjects are weak signals
- Ignore administrative tags

**User Prompt Components:**
1. Book metadata (title, author, description, OL subjects max 15)
2. Tier-specific instruction with word count emphasis
3. Vibe examples (3-6 per tier)
4. Ontology with buckets/mappings
5. Output requirements
6. Subject quality rules
7. Catalog handling guidance
8. Word count reminder

**Vibe Examples:**

RICH (8-14 words):
- "Sweeping historical saga exploring love and loss across generations" (11)
- "Dark psychological thriller unraveling secrets in a coastal town" (10)

SPARSE (4-8 words):
- "Thought-provoking exploration of wartime ethics" (5)
- "Gripping detective story with noir atmosphere" (6)

**Per-Tier Tracking:**
- First-attempt success rate

- Average latency

### Fingerprinting

**Hash Components:**
- item_idx
- title (normalized: lowercased, stripped)
- author (normalized)
- subjects (sorted, normalized)
- **tone_names** (sorted, normalized) ← RESOLVED from tone_ids
- **genre_name** (normalized) ← RESOLVED from genre_id
- vibe (normalized)
- embed_version
- tags_version

**Algorithm:**
- Create normalized dict with all components
- Serialize to canonical JSON (sorted keys)
- Compute SHA256 hex digest

**Skip Logic:**
- Query existing fingerprint from DB
- Compute current fingerprint
- If match: skip (unchanged)
- If different or missing: embed (changed or new)

### CRITICAL: Tone/Genre Resolution

**Problem:**
- Events store tone **IDs** (integers: 5, 12, 18)
- Events store genre **ID** (integer: 5) and genre **slug** ("science-fiction")
- Embeddings need display **names** for semantic meaning

**MUST RESOLVE BEFORE EMBEDDING:**

**Tone Resolution:**
- Load appropriate ontology CSV (v1 or v2)
- Build map: `tone_id → display_name`
- Example: `5 → "Dark"`, `12 → "Fast-paced"`

**Genre Resolution:**
- Load genres CSV
- Build map: `genre_id → display_name`
- Example: `5 → "Historical Fiction"`, `3 → "Literary Fiction"`
- Use `genre_id` from event (more direct than using slug)

**Ontology Loader:**
- Initialize once at worker startup
- Support both v1 and v2 tones
- Handle missing IDs gracefully (log warning)
- Cache in memory

### Embedding Text Format

**Template:**
```
{title} — {author} | genre: {genre_name} | subjects: {s1}, {s2}, ... | tones: {t1}, {t2}, ... | vibe: {vibe}
```

**Rules:**
- Title/author: always included
- Genre: **resolved name** (not slug)
- Subjects: comma-separated
- Tones: **resolved names** (not IDs)
- Vibe: may be empty for MINIMAL/BASIC
- Separator: ` | `

**Example:**
```
The Three-Body Problem — Liu Cixin | genre: Science Fiction | subjects: alien contact, astrophysics, Cultural Revolution, Chinese history, hard science fiction | tones: Dark, Cerebral, Epic | vibe: Mind-bending hard sci-fi exploring humanity's first contact with alien civilization
```

**Empty Field Handling:**
- No vibe: omit vibe section
- No tones: omit tones section
- No subjects: omit subjects section (rare)
- Genre always required

### Embedding Worker

**Architecture:**
- Standalone daemon (separate from enrichment)
- Components:
  - OntologyResolver (load CSVs, resolve IDs/slugs)
  - EnrichmentFetcher (query SQL for enriched books)
  - FingerprintTracker (manage SQLite DB)
  - EmbeddingClient (call API with retry)
  - AccumulatorWriter (append .npz batches)
  - CoverageMonitor (update metadata.json)

**Operation Modes:**

**Continuous:**
- Run every N minutes (default 15-30)
- Process all books needing embeddings
- Reaches steady state (most runs skip everything)
- Production mode

**One-Shot:**
- Process everything once, exit
- For initial 250k load
- Progress reporting with ETA
- Testing mode

**Targeted:**
- Process specific item_idx list
- For backfills/corrections

**Processing Flow:**

1. **Startup**
   - Load ontology resolvers
   - Connect to fingerprint DB
   - Load accumulator metadata
   - Verify directories

2. **Fetch Phase**
   - Query SQL for enriched books
   - Join: books, book_genres, book_tones, book_llm_subjects, book_vibes
   - Aggregate subjects and tone_ids per book

3. **Filter Phase (per book)**
   - Resolve tone_ids → tone_names
   - Resolve genre_slug → genre_name
   - Compute current fingerprint
   - Query existing fingerprint
   - If match: skip
   - If different/missing: queue for embedding

4. **Batch Phase**
   - Group into batches of 256
   - Build embedding texts (with resolved names)
   - Call embedding API
   - Write batch to .npz
   - Update fingerprint DB
   - Update metadata.json

5. **Reporting**
   - Log: X embedded, Y skipped, Z errors
   - Update coverage stats by tier
   - Calculate throughput

**EnrichmentFetcher Query:**
- Single query with LEFT JOINs
- GROUP BY book (aggregate subjects/tones)
- Filter: WHERE genre IS NOT NULL
- Return flattened dicts

**EmbeddingClient Specs:**
- Model: sentence-transformers/all-MiniLM-L6-v2 (384 dim)
- Batch size: 256 texts/call
- Retry: 3 attempts, exponential backoff
- Timeout: 30s/batch
- Rate limiting: respect API quotas

**AccumulatorWriter:**
- Append-only (never modify existing)
- Atomic writes (temp file → rename)
- Sequential numbering (batch_00000.npz)
- Compression: numpy savez_compressed

**Initial Load Strategy:**

**Approach: Single One-Shot Run**
- Process 250k books in ~256-book batches
- Expected: 3-5 hours (depends on API limits)
- Progress every 50 batches
- Periodic metadata.json updates

**Parallelization (Optional):**
- Multiple workers, subset item_idx ranges
- Shared fingerprint DB with locking
- Coordinate via run_id
- If initial load needs acceleration

**Steady State:**
- Continuous mode every 15 min
- Most runs skip most books (fingerprints match)
- Typical: 0-50 new embeddings/run
- Accumulator grows slowly

### Index Building

**Trigger Conditions:**
- Coverage threshold reached (95%, 98%, 99%)
- Scheduled cadence (weekly, bi-weekly)
- After large backfill
- On-demand

**Build Process:**

1. **Load Batches**
   - Read all batch_*.npz files
   - Load vectors and ids

2. **Deduplicate**
   - Map: item_idx → (vector, timestamp)
   - For duplicates: keep latest
   - Result: single vector per book

3. **Consolidate**
   - Create numpy arrays: ids, vectors
   - Ensure alignment

4. **Build FAISS Index**
   - Type: HNSW
   - Params: M=32, efConstruction=200
   - Time: 10-30 min for 250k

5. **Quality Checks**
   - Dimensionality (384)
   - No NaN/Inf
   - Vector norms reasonable
   - Sample queries work
   - No duplicate IDs

6. **Save Build**
   - Dated directory: build_YYYYMMDD_HHMMSS
   - Files: index.faiss, vectors.npy, ids.npy
   - manifest.json (metadata)
   - quality_report.json (QC results)

7. **Archive (Optional)**
   - Archive old accumulator batches
   - Keep recent (7 days)
   - Fingerprint DB remains

## Phase 4: Enhanced Quality (FUTURE - OPTIONAL)

### Two-Stage Enrichment

**Concept:** Two LLM calls for quality at 2x cost

**Stage 1: Broad Extraction**
- Generate 12-15 subject candidates
- Multiple genre/tone options
- Long vibe with alternatives

**Stage 2: Critical Refinement**
- Select best 8 subjects
- Finalize single genre
- Select best 2-3 tones
- Polish vibe

**Decision:**
- A/B test on 10% RICH tier
- Measure improvement in:
  - Subject specificity
  - Vibe quality
  - Search relevance
- If >10% improvement: consider rollout
- If <5%: not worth cost

### Hierarchical Tone Selection

**Enhancement:** Multi-level process

**Step 1: Bucket Selection**
- LLM picks 1-3 relevant buckets
- Example: "PACING and MOOD"

**Step 2: Tone Selection**
- Select from chosen buckets only
- Example: PACING→"fast-paced", MOOD→"dark, tense"

**Benefits:**
- Reduces cognitive load
- Encourages diversity
- More systematic

### Few-Shot Examples

**Enhancement:** Include 2-3 exemplars in prompt

**Example Structure:**
```
Example 1 (RICH, Fiction):
Book: "The Name of the Wind"
Subjects: [coming-of-age, magic systems, unreliable narrator, ...]
Tones: [lyrical, epic, mysterious]
Genre: fantasy
Vibe: "Lyrical epic fantasy following a legendary hero's formative years"

Example 2 (RICH, Non-fiction):
Book: "Sapiens"
Subjects: [human evolution, cognitive revolution, ...]
Tones: [accessible, thought-provoking, sweeping]
Genre: history
Vibe: "Sweeping narrative exploring humanity's journey from stone age to space age"
```

**Benefits:**
- Shows quality level
- Demonstrates specificity
- Covers fiction/non-fiction

**Considerations:**
- Adds 200-300 tokens/example
- May bias toward examples
- Rotate to avoid anchoring

### Noisy Subjects Integration

**Current:** Disabled in prompts

**Enhancement:** Re-enable with validation

**Approach:**
- Pass OL subjects to LLM
- Instruction: "Validate these, keep relevant, add 3-5 more"
- Better than ignoring weak signals

**Benefits:**
- Leverages existing metadata
- LLM filters quality
- May improve coverage

### Semantic Deduplication

**Post-processing enhancement**

**Approach:**
- After LLM generates subjects
- Embed all proposed subjects
- Calculate pairwise similarities
- Remove near-duplicates (>0.85 threshold)

**Example:**
- Input: ["Greek mythology", "ancient Greek deities"]
- Output: ["Greek mythology"] (removed duplicate)

**Benefits:**
- Catches semantic duplicates
- Improves subject distinctiveness

### Cross-Field Validation

**Quality check enhancement**

- Flag mismatch: dark/somber tones + comedy genre
- Flag generic: all subjects broad, vibe template-like
- Route flagged cases for review/re-enrichment

**Benefits:**
