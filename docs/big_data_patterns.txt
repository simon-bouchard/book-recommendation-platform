Enrichment jobs currently output JSONL files with validated fields (subjects, tone_ids, genre, vibe, tags_version). These files serve as the bronze layer. Apache Spark reads the compacted JSONL (keyed by item_idx), separates valid vs. error rows, and transforms the data into normalized relational tables (book_subjects, book_tones, book_genres, book_vibes). Spark writes these results into the silver layer (structured storage for training/analytics) and supports further processing into gold features (subject embeddings, Bayesian stats, etc.).

The roadmap adds Kafka as the ingestion layer. Enrichment jobs will publish results as events (enrich.results) instead of, or alongside, file writes. Spark Structured Streaming (or a Kafka sink) will persist raw events into object storage as the bronze layer. Spark will then consume from Kafka topics rather than JSON files, apply the same validation and normalization, and write to SQL and Delta/Iceberg. Kafka will also provide dead-letter topics for invalid payloads, replay support, and compaction for re-enrichment keyed by (item_idx, tags_version).

This design unifies batch and streaming: Spark is the transformation and load engine, while Kafka provides transport, buffering, retries, and real-time ingestion.
