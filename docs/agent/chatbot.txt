Context and System Overview
You are tasked with migrating a FastAPI-based book recommendation chatbot system from ReAct agents to LangGraph agents while applying clean architecture principles. The system currently has:

A working FastAPI backend with sophisticated ML recommendation models
ReAct-based agents that parse tool execution steps in a brittle way
Mixed concerns where HTTP endpoints handle business logic
Complex book ID extraction from agent execution steps
Multiple agent types: recommendation, web search, docs, and response agents

Current Pain Points:

Book recommendations aren't properly extracted from LangGraph agents
Complex step-by-step parsing that breaks easily
Mixed abstractions in the agent system
Tight coupling between HTTP layer and agent execution

Migration Goal: Refactor only the app/agents/ folder to use LangGraph with clean architecture, while keeping all other systems (routes, models, database, etc.) unchanged.
Current Agent Folder Structure
app/agents/
├── base.py                    # BaseLLMAgent (ReAct-based)
├── branches/                  # Individual agent implementations
├── orchestrator/              # Conductor and Router
├── tools/                     # Tool registry and implementations
├── prompts/                   # System prompts
├── runtime.py                 # Book ID extraction utilities
├── schemas.py                 # Data schemas
├── settings.py                # Configuration
└── logging.py                 # Logging utilities
Phase Instructions for LLM Implementation
Phase 1: Agent Domain Layer Design
Task: Create domain entities and interfaces for the agent system that separate business logic from infrastructure concerns.
Requirements:

Design clean domain entities for agent execution state, requests, and results
Create abstract interfaces that define what an agent should do (not how)
Design a result processor that handles book ID extraction logic cleanly
Ensure backward compatibility with existing AgentResult schema
Create value objects for agent configuration and capabilities

Key Design Principles:

Domain entities should not depend on LangGraph or any specific framework
Business rules for book recommendation processing should be explicit
Agent interfaces should be technology-agnostic
Result processing should be deterministic and testable

Deliverables:

Domain entity classes for agent execution
Abstract agent interface
Result processor service
Configuration value objects
Clear separation between domain and infrastructure concerns

Phase 2: LangGraph Infrastructure Implementation
Task: Create LangGraph-based agent implementations that replace the current ReAct system while maintaining the same external interface.
Requirements:

Build a base LangGraph agent class that implements the domain interface
Design proper state management for LangGraph workflows (not step-based)
Create specific implementations for each agent type (recsys, web, docs, response)
Implement proper error handling and recovery within graph execution
Ensure deterministic book ID extraction from final states (not steps)

Technical Constraints:

Must work with existing tool implementations
Should handle async execution properly
Must maintain compatibility with current Conductor interface
Error states should be recoverable and logged appropriately

Agent-Specific Requirements:

Recommendation Agent: Multi-node workflow with analysis, search, curation, and response nodes
Web Agent: Search and synthesis workflow with external API calls
Docs Agent: Simple lookup and formatting workflow
Response Agent: Single-node conversational response

Deliverables:

Base LangGraph agent class
Four specific agent implementations
State management patterns
Error handling strategies
Integration with existing tool system

Phase 3: Tool System Modernization
Task: Simplify the tool system by removing LangChain wrapper complexity while maintaining all existing functionality.
Requirements:

Replace LangChain Tool wrappers with native function interfaces
Maintain all existing tool gating and permission logic
Simplify tool argument handling (remove complex string/JSON parsing)
Create a cleaner tool manager that works with LangGraph
Preserve all existing tool implementations (semantic search, ALS, etc.)

Migration Strategy:

Keep existing tool functions but remove wrapper layers
Maintain backward compatibility during transition
Simplify input/output handling
Preserve security and access control features

Deliverables:

Simplified tool manager class
Native function interfaces for all tools
Preserved gating and permission system
Clean integration with LangGraph agents

Phase 4: Orchestrator Integration
Task: Modify the Conductor to work with new LangGraph agents while maintaining exact API compatibility with existing routes.
Requirements:

Create an agent factory that instantiates appropriate LangGraph agents
Maintain existing Conductor interface (same method signatures)
Implement feature flags for gradual migration
Preserve existing router functionality
Ensure proper context passing to agents

Backward Compatibility:

Same input parameters to Conductor.run()
Same output format (AgentResult)
No changes required to routes/chat.py
Gradual rollout capability with environment flags

Deliverables:

Agent factory with proper dependency injection
Modified Conductor with feature flag support
Preserved router integration
Migration strategy implementation

Phase 5: Migration and Validation Strategy
Task: Design and implement a safe migration approach that allows testing and rollback.
Requirements:

Feature flag system for enabling new agents individually
Comprehensive testing strategy for each agent type
Performance monitoring and comparison tools
Rollback procedures if issues arise
Validation that book recommendations work correctly

Safety Measures:

A/B testing capability between old and new agents
Detailed logging for comparison
Automated tests for each agent type
Performance benchmarks
User experience validation

Monitoring Requirements:

Track book ID extraction success rates
Monitor response times and error rates
Compare recommendation quality between systems
Log any compatibility issues

Deliverables:

Feature flag implementation
Testing framework for agents
Migration runbook
Monitoring and alerting setup
Rollback procedures

Implementation Guidelines for Each Phase
Code Quality Standards:

Follow SOLID principles throughout
Use dependency injection for testability
Implement proper error handling and logging
Write comprehensive docstrings and type hints
Create unit tests for domain logic

Architecture Principles:

Keep domain logic separate from infrastructure
Use interfaces to define contracts
Avoid circular dependencies
Maintain single responsibility for each class
Ensure loose coupling between components

Migration Safety:

Never break existing APIs during migration
Test each phase thoroughly before proceeding
Maintain backward compatibility throughout
Have rollback plans for each phase
Monitor system health continuously

Your implementation should result in a cleaner, more maintainable agent system that properly handles book recommendations while preserving all existing functionality and improving the overall code quality.
