# Chatbot Testing Implementation Plan: Mock-Based Architecture

## Executive Summary

### Problem Statement

Current Phase 1 tests (33 Conductor integration tests) use real LLM agents to validate infrastructure concerns:
- **Expensive**: ~$0.50-2.00 per test run (33 LLM calls)
- **Slow**: 2-5 minutes execution time
- **Non-deterministic**: LLM variance causes flaky tests
- **Unclear scope**: Tests mix infrastructure validation with LLM quality evaluation
- **CI unfriendly**: Cannot run on every commit due to cost/time

This violates the principle of **separation of concerns** - infrastructure tests should not depend on LLM behavior.

### Solution Architecture

Implement a two-tier testing strategy:

1. **Infrastructure Tests** (Phase 1.5 + Phase 2): Use mocked agents/LLMs
   - Fast (<30 seconds total)
   - Free (no API calls)
   - Deterministic (same results every time)
   - Test orchestration, data flow, error handling
   - Run on every commit

2. **Evaluation Tests** (Phase 3 - documentation only): Use real LLMs
   - Slow (minutes)
   - Expensive (API costs)
   - May have variance
   - Test LLM output quality, agent behavior
   - Run pre-deploy or on-demand

### Phases Overview

|-------|-------|-------|------|--------|
| Phase 1.5 | Refactor Conductor tests to use mocks | 33 tests (modified) | 2-2.5h | **New** |

## Architecture Decisions

### Decision 1: Two-Tier Testing Strategy

**Approach:** Separate infrastructure tests (mocked) from evaluation tests (real LLMs).

**Rationale:**
- **Infrastructure concerns**: Routing logic, data flow, error handling, parameter propagation
  - Don't need real LLMs - just verify "does the right thing get called with right params?"
  - Should be fast, free, deterministic
  - Run frequently (every commit)

- **Quality concerns**: LLM output accuracy, relevance, helpfulness
  - Need real LLMs - evaluating actual behavior
  - Can be slow, expensive, variable
  - Run infrequently (pre-deploy, on-demand)

**Alternative rejected:** Keep all tests using real LLMs
- Makes infrastructure tests slow and expensive
- Couples infrastructure testing to LLM API availability
- Mixes concerns (what is the test actually validating?)

### Decision 2: Mock at Agent Boundary, Not Tool Boundary

**Approach:** Mock entire agents (RecommendationAgent, WebAgent, etc.), not individual tools or LLM calls.

**Rationale:**
- Tests Conductor's responsibility: orchestrating agents
- Doesn't test agent internals (those have separate tests)
- Clean boundary - agents are clear integration points
- Easier to maintain - agent interface more stable than tool internals

**Contract:**
```python
# All agents implement this interface
class BaseAgent:
    def execute(self, request: AgentRequest) -> AgentResponse:
        """Execute agent logic and return response."""
        pass
```

**Alternative rejected:** Mock tools/LLM calls within agents
- Too granular - couples tests to agent implementation details
- Would break when agent refactors tools internally
- Harder to maintain - many more mock points

### Decision 3: Fixture-Based Mock Injection

**Approach:** Create pytest fixtures that return pre-configured mocked agents.

**Rationale:**
- Consistent mock behavior across all tests
- Easy to customize per test if needed
- Follows pytest best practices
- Reduces test setup code duplication

**Pattern:**
```python
@pytest.fixture
def mock_agent_factory(mock_recsys_agent, mock_web_agent, ...):
    """Returns factory that creates mocked agents."""
    factory = Mock()
    factory.create_agent = lambda agent_type, **kwargs: {
        "recsys": mock_recsys_agent,
        "web": mock_web_agent,
        # ...
    }[agent_type]
    return factory

def test_something(mock_agent_factory):
    conductor = Conductor()
    conductor.factory = mock_agent_factory  # Inject mocks
    result = conductor.run(...)  # Uses mocks
```

**Alternative rejected:** Mock directly in each test
- 33 tests × multiple agents = massive duplication
- Inconsistent mock behavior across tests
- Hard to maintain when agent interface changes

### Decision 4: Fixture Organization Strategy

**Approach:** Three-level fixture hierarchy.

**Structure:**
```
tests/integration/chatbot/
├── conftest.py                    # Level 1: Generic (db_session)
├── recsys_fixtures.py             # Level 2: Domain-specific (user fixtures)
│
├── conductor/
│   └── conftest.py                # Level 3: Test-specific (mock agents)
│
└── agents/recommendation/
    └── conftest.py                # Level 3: Test-specific (mock sub-agents)
```

**Rationale:**
- **Level 1 (Generic)**: Fixtures any test might need (database, environment)
- **Level 2 (Domain)**: Reusable domain fixtures (recsys users) - shared across multiple test suites
- **Level 3 (Test-specific)**: Mocks and fixtures specific to one test suite

**Why this matters:**
- Avoids fixture pollution (web agent tests don't inherit recsys fixtures)
- Clear ownership (where should new fixture go?)
- Reusability (recsys_fixtures used by both conductor and recommendation tests)

### Decision 5: Test Data Factories for Mocks

**Approach:** Create builder/factory classes to generate test data and configure mocks.

**Why:**
- **Builders**: Fluent API for configuring mocks (`builder.returns_strategy(...).build()`)
- **Factories**: Consistent test data generation (`CandidateFactory.create_batch(60)`)
- **Separation**: Builders for behavior, factories for data

**Pattern:**
```python
class MockAgentBuilder:
    def __init__(self):
        self.mock = Mock()
        # Default behavior
        self.mock.execute.return_value = AgentResponse(...)

    def returns_failure(self, error_msg: str):
        self.mock.execute.side_effect = RuntimeError(error_msg)
        return self  # Chain

    def returns_success(self, **response_kwargs):
        self.mock.execute.return_value = AgentResponse(**response_kwargs)
        return self  # Chain

    def build(self):
        return self.mock
```

**Alternative rejected:** Configure mocks directly in tests
- Repetitive setup code in every test
- Hard to maintain consistent behavior
- Tests become 80% setup, 20% assertions

## Phase 2: RecommendationAgent Pipeline Tests (3-3.5 hours)

### Overview

Add 20 integration tests that validate RecommendationAgent's 3-stage pipeline orchestration using mocked sub-agents.

**Focus:** How RecommendationAgent coordinates PlannerAgent → RetrievalAgent → CurationAgent.

**Dependency:** Phase 1.5 must be complete (provides patterns and fixtures to build on).

### Architecture

**What we're testing:**
```
RecommendationAgent (real)
├── PlannerAgent (mocked) → returns strategy
├── RetrievalAgent (mocked) → returns candidates based on strategy
└── CurationAgent (mocked) → returns final recommendations from candidates
```

**Why this design:**
- Tests RecommendationAgent's orchestration logic (its responsibility)
- Doesn't test sub-agent internals (those have separate tests)
- Fast, deterministic, maintainable

**Dependency Injection Contract:**
```python
class RecommendationAgent:
    def __init__(
        self,
        current_user: User,
        db: Session,
        user_num_ratings: int,
        allow_profile: bool = False,
        planner_agent: Optional[PlannerAgent] = None,      # Injectable
        retrieval_agent: Optional[RetrievalAgent] = None,  # Injectable
        curation_agent: Optional[CurationAgent] = None,    # Injectable
    ):
        self.planner = planner_agent or PlannerAgent(...)   # Use injected or default
        self.retrieval = retrieval_agent or RetrievalAgent(...)
        self.curation = curation_agent or CurationAgent(...)
```

**Note:** This already exists (verified in orchestrator.py lines 47-49, 86-109).

**Builder pattern example:**
```python
class MockPlannerBuilder:
    def __init__(self):
        self.mock = Mock()
        # Default behavior
        self._strategy = PlannerStrategy(
            tools=["als_recs"],
            subjects=None,
            reasoning="Default strategy"
        )

    def returns_strategy(self, tools: List[str], subjects: Optional[List[str]] = None):
        self._strategy = PlannerStrategy(
            tools=tools,
            subjects=subjects,
            reasoning=f"Using {', '.join(tools)}"
        )
        return self  # Chain

    def returns_failure(self, error_msg: str = "Planner failed"):
        self.mock.execute.side_effect = RuntimeError(error_msg)
        return self

    def build(self):
        if not self.mock.execute.side_effect:
            self.mock.execute.return_value = self._strategy
        return self.mock
```

## Risk Management

### Risk 1: Refactoring breaks existing tests

**Mitigation:**
- Refactor one file at a time
- Run tests after each file
- Keep git commits small (easy to revert)

**Fallback:**
- If refactor too complex, pause and reassess
- Can keep some tests with real agents temporarily

### Risk 2: Mocks don't match real agent behavior

**Mitigation:**
- Use actual agent interfaces (AgentRequest/AgentResponse schemas)
- Keep mocks simple (return valid data, don't emulate complex logic)
- Rely on evaluation tests to catch behavior mismatches

**Note:** This is acceptable - infrastructure tests validate orchestration, evaluation tests validate behavior.

### Risk 3: Test fixtures become too complex

**Mitigation:**
- Use builder pattern (fluent, self-documenting)
- Keep fixtures focused (single responsibility)
- Document fixture usage in test docstrings

**Warning signs:**
- Fixtures with 10+ parameters
- Tests spend more time setting up than asserting
- Hard to understand what test is checking

### Risk 4: Mock setup is inconsistent across tests

**Mitigation:**
- Centralize mock creation in fixtures
- Use builders to enforce consistency
- Review tests for duplicate setup code

**Pattern to avoid:**
```python
# Bad - inconsistent mock setup in every test
mock = Mock()
mock.execute.return_value = ...
```

**Pattern to use:**
```python
# Good - consistent fixture-based setup
agent = mock_orchestrator()  # Uses fixture
```

## Common Patterns & Anti-Patterns

### Pattern 1: Fluent Builder API

**Good:**
```python
mock = builder.returns_strategy(["als_recs"]).returns_candidates(60).build()
```

**Bad:**
```python
mock = Mock()
mock.execute.return_value = ...
# (repeated in every test)
```

### Pattern 2: Fixture-Based Agent Creation

**Good:**
```python
def test_something(mock_orchestrator):
    agent = mock_orchestrator()  # One line
    result = agent.execute(request)
```

**Bad:**
```python
def test_something(db_session, test_user):
    planner = Mock()
    retrieval = Mock()
    curation = Mock()
    agent = RecommendationAgent(..., planner, retrieval, curation)
    # (repeated setup in every test)
```

### Pattern 3: Verify Data Flow, Not Just Calls

**Good:**
```python
retrieval_input = mock_retrieval.execute.call_args[0][0]
assert retrieval_input.strategy.tools == ["als_recs"]
assert retrieval_input.query == "recommend books"
```

**Bad:**
```python
assert mock_retrieval.execute.called  # Weak - doesn't test data
```

### Pattern 4: Realistic Test Data via Factories

**Good:**
```python
candidates = CandidateFactory.create_batch(60)
mock.execute.return_value = candidates
```

**Bad:**
```python
candidates = [
    Candidate(item_idx=1, title="Book 1", ...),
    Candidate(item_idx=2, title="Book 2", ...),
    # ... 58 more lines ...
]
```

## Appendix A: Key Schemas & Contracts

### AgentRequest (Input to agents)
```python
@dataclass
class AgentRequest:
    query: str                          # Current user query
    history: List[dict]                 # Conversation history [{"u": ..., "a": ...}]
    context: dict                       # Additional context
        # context keys:
        # - profile_allowed: bool
        # - user_num_ratings: int
        # - conv_id: str
        # - uid: int
    user: Optional[User]                # Current user object
    db: Optional[Session]               # Database session
```

### AgentResponse (Output from agents)
```python
@dataclass
class AgentResponse:
    text: str                           # Response text (required)
    success: bool                       # Whether execution succeeded
    book_ids: Optional[List[int]]       # For recsys agent
    policy_version: str                 # Policy version (required)
    execution_state: Optional[dict]     # Metadata, tool calls, etc.
```

### AgentResult (Final output from Conductor)
```python
@dataclass
class AgentResult:
    text: str                           # Response text
    success: bool                       # Overall success
    book_ids: Optional[List[int]]       # Book recommendations
    policy_version: str                 # Policy version
    # ... other fields from AgentResponse
```

### PlannerStrategy (Output from PlannerAgent)
```python
@dataclass
class PlannerStrategy:
    tools: List[str]                    # Tools to use (e.g. ["als_recs"])
```

### Candidate (Output from RetrievalAgent)
```python

class Candidate:
```

```python
@dataclass
class BookRecommendation:
```

## Questions & Clarifications

**Q: Why not mock the router in Phase 1.5?**
A: Router is already tested separately in evaluation tests. Conductor tests verify "does the router's decision get executed correctly?", not "does the router make the right decision?". We can mock router when testing error handling.

**Q: Can we skip Phase 3 (documentation)?**
A: Not recommended. Without clear documentation, future tests may end up in the wrong place, mixing infrastructure and evaluation concerns again.

**Q: What if refactoring Phase 1.5 takes longer than estimated?**
A: Can pause after each test file and reassess. Each file is independent. If one file is particularly complex, can defer to Phase 2 completion.

**Q: Should we add type hints to all test code?**
A: Yes for fixtures and helpers (improves IDE support). Optional for test methods (docstrings are more important).

**Q: How do we handle flaky tests if they appear?**
A: Mocked tests should not be flaky. If a test is flaky, it likely indicates:
1. Test pollution (state leaking between tests)
2. Mock not properly configured
3. Test has timing dependency (use pytest-timeout)

**Q: Can we parallelize test execution?**
A: Yes, use `pytest -n auto` (requires pytest-xdist). Should work since tests are independent. Might need to ensure db session fixtures are thread-safe.
