**Goal:** Add 20 integration tests that validate RecommendationAgent's 3-stage pipeline (Planner → Retrieval → Curation) orchestration logic using mocked sub-agents.

## Architecture Decisions

### Decision 1: Mock Strategy
**Approach:** Mock the 3 sub-agents (PlannerAgent, RetrievalAgent, CurationAgent), not their internals.

**Why:**
- Tests orchestration logic (RecommendationAgent's responsibility)
- Avoids testing sub-agent implementations (those have separate tests)
- Fast execution (no LLM calls)
- Deterministic (no flaky tests)

**Alternative rejected:** Mock individual tools/LLM calls
- Too granular for integration tests
- Couples tests to implementation details
- Would break when sub-agents change internally

### Decision 2: Test Helpers Architecture
**Approach:** Create builder classes for mocks + factory classes for test data.

**Why:**
- **Builders:** Fluent API makes tests readable (`builder.returns_strategy(...).build()`)
- **Factories:** Consistent test data without magic numbers
- **Separation:** Builders for mocks, factories for data (different concerns)
- **Reusability:** All 20 tests use same helpers

**Structure:**
```
MockPlannerBuilder    → builds configured mock PlannerAgent
MockRetrievalBuilder  → builds configured mock RetrievalAgent
MockCurationBuilder   → builds configured mock CurationAgent
CandidateFactory      → creates Candidate test data
BookFactory           → creates BookRecommendation test data
```

**Alternative rejected:** Mock directly in each test
- 20 tests × 3 mocks = 60 mock setups (massive duplication)
- Hard to maintain consistent test data
- Tests become 80% setup, 20% assertions

### Decision 3: Fixture Strategy
**Approach:** Add `mock_orchestrator` fixture that returns a function to create agents with mocks.

**Why:**
- One-line agent creation in tests: `agent = mock_orchestrator()`
- Easy customization: `agent = mock_orchestrator(planner=custom_mock)`
- Consistent user/db setup across all tests
- Compatible with Phase 1 fixtures

**Usage pattern:**
```python
def test_something(mock_orchestrator, mock_planner_builder):
    # Customize one mock
    custom_planner = mock_planner_builder.returns_strategy(...).build()

    # Create agent with custom planner, default retrieval/curation
    agent = mock_orchestrator(planner=custom_planner)

    # Test
    response = agent.execute(request)
    assert ...
```

**Alternative rejected:** Create agent manually in each test
- Repetitive setup code in every test
- Easy to forget parameters (user, db, num_ratings, etc.)
- Harder to maintain when agent constructor changes

### Decision 4: Test Categories
**Approach:** Organize 20 tests into 4 logical categories as separate test classes.

**Categories:**
1. **Stage Transitions (6 tests):** Data flows correctly between stages
2. **Parameter Propagation (4 tests):** Warm/cold user, profile access, etc.
3. **Error Handling (6 tests):** Graceful failure at each stage
4. **Full Pipeline Flow (4 tests):** End-to-end realistic scenarios

**Why this organization:**
- Each category tests a different aspect of orchestration
- Easy to run subsets during development
- Clear responsibility boundaries
- Matches mental model of pipeline operation

**Alternative rejected:** All tests in one class
- Harder to navigate 20 tests
- Unclear what each test category covers
- Can't easily run "just parameter tests"

#### Task 2: Test Helpers (30 min)
**File:** `tests/integration/chatbot/test_helpers.py`

**What to implement:**

1. **MockPlannerBuilder class:**
   - `returns_strategy(tools, reasoning, profile_data)` → self
   - `raises_error(exception)` → self
   - `build()` → Mock
   - Default: returns basic strategy if nothing configured

2. **MockRetrievalBuilder class:**
   - `returns_candidates(candidates, tools_used)` → self
   - `returns_empty()` → self
   - `raises_error(exception)` → self
   - `build()` → Mock
   - Default: returns 60 candidates

3. **MockCurationBuilder class:**
   - `returns_recommendations(books, prose)` → self
   - `returns_failure(error_message)` → self
   - `raises_error(exception)` → self
   - `build()` → Mock
   - Default: returns 5 books

4. **CandidateFactory class:**
   - `create(item_idx, title, author, subjects, tones, vibe)` → Candidate
   - `create_batch(count, base_idx)` → List[Candidate]

5. **BookFactory class:**
   - `create(item_idx, title, author, explanation)` → BookRecommendation
   - `create_batch(count, base_idx)` → List[BookRecommendation]

**Key design point:** Builders use method chaining for readability.

**Verification:**
```python

# Can create test data
```

## Critical Design Patterns

### Pattern 1: Builder Fluent API
```python
# Good - readable, chainable
mock = builder.returns_strategy(["als_recs"]).build()

# Bad - verbose, error-prone
mock = Mock()
mock.execute.return_value = PlannerStrategy(...)
```

### Pattern 2: Factory Batch Creation
```python
# Good - consistent, no magic numbers
candidates = CandidateFactory.create_batch(60)

# Bad - repetitive, error-prone
candidates = [
    Candidate(item_idx=10000, title="Book 1", ...),
    Candidate(item_idx=10001, title="Book 2", ...),
    # ... 58 more ...
]
```

### Pattern 3: Fixture-Based Agent Creation
```python
# Good - one line, clear intent
agent = mock_orchestrator(planner=custom_planner)

# Bad - repetitive setup in every test
agent = RecommendationAgent(
    current_user=test_user,
    db=db_session,
    user_num_ratings=10,
    allow_profile=False,
    planner_agent=custom_planner,
    retrieval_agent=mock_retrieval,
    curation_agent=mock_curation,
)
```

### Pattern 4: Mock Call Verification
```python
# Good - verify what was passed
planner_input = mock_planner.execute.call_args[0][0]
assert planner_input.query == "recommend books"

# Bad - verify it was called (doesn't test data flow)
assert mock_planner.execute.called
```

## Common Pitfalls and Solutions

### Pitfall 1: Mock Not Being Used
**Symptom:** Test passes but mock.execute.called is False

**Solution:** Verify agent was created with `mock_orchestrator(planner=your_mock)`

### Pitfall 2: Fixtures Not Found
**Symptom:** "fixture 'mock_orchestrator' not found"

**Solution:** Check fixture is in conftest.py, not in test file

### Pitfall 3: Wrong Call Args Index
**Symptom:** KeyError or IndexError when checking call_args

**Solution:**
- Positional args: `call_args[0][index]`
- Keyword args: `call_args[1]['name']`
- Alternative: `call_args.args[0]` or `call_args.kwargs['name']`

### Pitfall 4: Test Passes but Tests Nothing
**Symptom:** Test never fails even when breaking code

**Solution:**
- Add assertion that mock was called: `assert mock.execute.called`
- Verify specific data: `assert input.query == expected`
- Don't just check types: `assert isinstance(x, Y)` is weak

### Pitfall 5: Builders Don't Chain
**Symptom:** `returns_strategy().returns_strategy()` doesn't work

**Solution:** All builder methods must `return self`

## Maintenance Guide

### Adding New Tests
1. Choose appropriate test class (Stage/Parameter/Error/Flow)
2. Use existing fixtures and builders
3. Follow naming: `test_<what>_<condition>`
4. Add docstring explaining what's tested
5. Run just your test: `pytest path/to/test.py::TestClass::test_name -v`

### Updating for Code Changes
- **Orchestrator changes:** May need new assertions in Stage Transitions tests
- **New parameters:** Add tests to Parameter Propagation category
- **New failure modes:** Add tests to Error Handling category
- **Schema changes:** Update factories/builders

### When Tests Fail
1. Run single failing test with `-vv` for details
2. Check if mock was called: `print(mock.execute.called)`
3. Inspect call args: `print(mock.execute.call_args)`
4. Verify fixture setup: Add `print(agent)` before execute
5. Check Phase 1 still passes (may be orchestrator bug, not test bug)
