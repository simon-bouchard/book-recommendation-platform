Chatbot Multiagent System - Testing Plan
System Architecture

Router: LLM-based classifier that routes queries to appropriate agents
Agents: Recsys (complex), Web Search, Documentation, Response (simple conversational)
Flow: User query → Router → Selected Agent → Response

Testing Philosophy

Use real LLM for intelligence/decision-making (routing, intent understanding, query comprehension)
Mock tools/dependencies for testing logic/flow/error handling
Balance coverage with cost and speed
Test layers independently before testing integration

Test Layers
1. Router Tests (Priority: CRITICAL)
Type: Integration tests with real LLM
Count: 15-20 representative queries
What to test:

Classification accuracy for each agent type
Edge cases (ambiguous queries, multi-intent, out-of-scope)
Routing consistency with temperature=0

Why real LLM: Router's core function IS the LLM classification - mocking tests nothing
2. Agent Structure Tests (Priority: HIGH)
Type: Unit tests with mocked tools/dependencies
Count: 40-60 tests across all agents
What to test:

Tool calling patterns and sequences
State transitions in LangGraph nodes
Error handling and recovery
Output structure and data validation
Edge case handling (empty results, timeouts, etc.)

Why mocked: Tests orchestration logic independent of LLM/tool variability
3. Agent Intelligence Tests (Priority: MEDIUM)
Type: Integration tests with real LLM, mocked/simplified tools
Count: 20-30 tests across all agents
What to test:

Query understanding and intent extraction
Appropriate tool selection for different query types
Decision-making logic within agent workflows
Response coherence and relevance

Why real LLM: Agent decision-making requires actual intelligence
4. Agent + Tools Integration Tests (Priority: MEDIUM)
Type: Integration tests with real tools, limited data
Count: 15-25 tests per complex agent
What to test:

Agent works correctly with actual tool implementations
Data flows properly through tool interfaces
Results are processed and formatted correctly
Specific focus on Recsys agent (most complex tool chain)

Why real tools: Catches interface mismatches and integration bugs
5. End-to-End Pipeline Tests (Priority: LOW-MEDIUM)
Type: Full system tests
Count: 10-15 golden test cases
What to test:

Complete user journeys for each agent type
Multi-turn conversations
Critical regression scenarios
Performance benchmarks

Why minimal: Expensive, slow, harder to debug - use for smoke testing only
Agent-Specific Considerations
Response Agent (conversational):

Minimal testing needed
Focus on routing TO it correctly
Basic integration test for tone/appropriateness

Documentation Agent:

Test retrieval accuracy heavily
Verify citation/source attribution
Test out-of-scope query handling

Web Search Agent:

Test date-based trigger logic (post-2004 queries)
Verify search query formation
Test result extraction and formatting

Recsys Agent (most complex):

Test last, most comprehensive suite
Focus on tool chain orchestration
Verify book ID extraction
Test recommendation quality (may need LLM-as-judge)

LLM-as-Judge Usage
Use selectively for subjective quality only:

Recommendation relevance (Recsys, Web agents)
Factual accuracy from sources (Docs agent)
Answer completeness and coherence

Don't use for: Tool usage, output structure, routing, error handling (use assertions instead)
Test Distribution Target
Unit Tests (mocked):          40-60 tests   | Fast, cheap
Integration (real LLM):       35-50 tests   | Medium speed/cost
E2E (full pipeline):          10-15 tests   | Slow, expensive

Total: ~100-125 tests
Estimated full run: 3-5 minutes, <$1 in API costs
Implementation Order

Router classification tests (highest ROI)
Agent structure/logic tests (fast feedback loop)
Agent intelligence tests (validates decisions)
Tool integration tests (catches interface issues)
E2E golden paths (regression safety net)

Key Principles

Don't mock intelligence: Use real LLM for classification and decision-making
Do mock infrastructure: Tools, databases, external APIs for fast feedback
Test boundaries: Focus on component interfaces and contracts
Determinism where possible: Use temperature=0, assert on structure not exact content
Fast feedback first: Unit tests should run in seconds
Golden dataset: Maintain 10-15 representative queries for regression testing
