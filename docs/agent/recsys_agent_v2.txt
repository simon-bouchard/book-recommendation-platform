Recommendation System Refactoring - Implementation Plan (Phase 1)
Overview
This document outlines the refactoring plan to split the monolithic recommendation agent into a multi-stage pipeline with clear separation of concerns.
Core Architecture:
User Query
    ↓
PlannerAgent (analyze + decide strategy)
    ↓
CandidateGeneratorAgent (execute + gather)
    ↓
CurationAgent (filter + rank + prose)
    ↓
Final Response
Note: The decision to split CurationAgent into separate Ranking and Explainer components will be deferred until after Phases 1-3 are complete. We need to investigate whether curator ranking issues can be resolved through better prompts and candidate preparation before adding architectural complexity.

Phase 1: Tool Reorganization
Goal: Separate tools by responsibility and create clear boundaries between different tool types.
1.1 Tool Category Definitions
Context Tools (for Planner):

user_profile - Get user's favorite subjects
recent_interactions - Get recent ratings/interactions
Purpose: Inform strategy decisions, NOT retrieve books
Returns: User preference data (subjects, genres, recent activity)

Retrieval Tools (for Candidate Generator):

als_recs - Collaborative filtering recommendations
book_semantic_search - Semantic/vibe-based search
subject_id_search - Convert subject names to IDs
subject_hybrid_pool - Subject-based recommendations
popular_books - Bayesian-ranked popular books (NEW)
Purpose: Retrieve candidate books
Returns: Lists of book candidates with metadata

1.2 New Tool: popular_books
Purpose: Extract Bayesian average logic from subject_hybrid_pool into dedicated tool
Current Problem:

subject_hybrid_pool(fav_subjects_idxs=None) → returns popular books
subject_hybrid_pool(fav_subjects_idxs=[1,2,3]) → returns subject-based recs
Mixed responsibility makes tool selection unclear

Solution:

Create popular_books tool with single responsibility
subject_hybrid_pool ONLY handles subject-based retrieval
Makes LLM tool selection more intuitive

Tool Signature:
pythondef popular_books(top_k: int = 100) -> ToolResult
```

**Tool Description for LLM:**
```
Get popular books ranked by Bayesian average rating.
Use for cold users with vague queries when no other context available.
Returns books sorted by rating quality + popularity.

```
### 2.4 Decision Logic

**Query Type Classification:**

| Query Type | Indicators | Recommended Tools | Fallback Tools |
|-----------|-----------|-------------------|----------------|
| **Vague** | "recommend", "what to read", no specifics | `als_recs` (warm) OR `subject_hybrid_pool` (cold w/ profile) OR `popular_books` (cold no profile) | `popular_books`, `book_semantic_search` |
| **Descriptive** | Atmosphere/mood/tone words ("dark", "cozy", "atmospheric") | `book_semantic_search` | `subject_hybrid_pool`, `popular_books` |
| **Genre-specific** | Explicit genres ("mystery", "fantasy", "historical fiction") | `subject_id_search` → `subject_hybrid_pool` | `book_semantic_search` |
| **Complex** | Multiple requirements (genre + vibe + constraints) | `subject_id_search` → `subject_hybrid_pool` + `book_semantic_search` | `popular_books` |

**Profile Tool Decision Tree:**
```
Should call user_profile?
├─ allow_profile = False → NO
├─ Query is descriptive (has atmosphere/genre terms) → NO (enough info in query)
└─ Query is vague → YES (need context to inform strategy)
```

### 2.5 Implementation Approach

**Agent Type:** Simple LLM call (NOT ReAct loop)
- Single analysis pass
- May call 0-2 profile tools via tool binding
- Returns strategy JSON
- No iteration needed

**LLM Configuration:**
- Tier: "medium" (strategy is simpler than execution)
- Temperature: 0.1 (want consistent strategy)
- JSON mode: Yes (for strategy output)
- Tool binding: Only if allow_profile=True

**Prompt Structure:**
```
[System prompt with strategy logic]

[Profile data: {profile_data}]  # If tools were called

Determine the best retrieval strategy.
Return JSON with recommended_tools, fallback_tools, reasoning.

Phase 3: CandidateGeneratorAgent
Goal: Execute retrieval tools and gather 60-120 candidate books
3.1 Responsibilities

Strategy Execution:

Receive strategy from Planner
Execute recommended tools with appropriate parameters
Accumulate book candidates with metadata


Quality Assessment:

After each tool call, inspect results
Check if books match query intent (subjects, genres, tones)
Detect relevance mismatches


Adaptive Fallback:

If primary tool returns irrelevant results → try fallback
If primary tool returns < 30 books → try additional tools
If primary tool excellent → stop early (60+ books)


Negative Constraint Handling:

Strip negative terms from semantic search queries
Example: "dark fantasy but no vampires" → search "dark fantasy"
Let Curator handle filtering



3.2 Input Schema
pythonCandidateGeneratorInput:
    query: str                          # Original user query
    strategy: PlannerStrategy           # Strategy from Planner
    profile_data: Optional[dict]        # Profile data if Planner gathered it
3.3 Output Schema
pythonCandidateGeneratorOutput:
    candidates: list[BookRecommendation]  # 60-120 books with full metadata
    tool_executions: list[ToolExecution]  # History of what was called
    reasoning: str                        # Why stopped, what was tried
pythonToolExecution:
    tool_name: str                      # Which tool
    arguments: dict                     # Parameters used
    book_count: int                     # How many books returned
    relevance_assessment: str           # "good match" | "off-target" | "mixed"

```

### 3.4 Decision Logic

**Tool Parameter Selection:**

| Tool | Parameter Decisions |
|------|-------------------|
| `als_recs` | `top_k=120` (fixed, always want many for curation) |
| `book_semantic_search` | `top_k=80-100`, `query=` structured format (genre/tones/subjects) |
| `subject_hybrid_pool` | `fav_subjects_idxs=` from profile or query, `top_k=100-120`, `weight=0.6-0.8` |
| `subject_id_search` | `phrases=` extracted genre terms, `top_k=5-10` |
| `popular_books` | `top_k=100-120` |

**Stopping Conditions:**
```
STOP if:
- Have 60+ books AND primary tool succeeded AND results are relevant
- Have 100+ books (regardless of quality - let Curator filter)
- Have tried primary + fallback AND both returned books
- Hit max iterations (7)

CONTINUE if:
- Under 30 books total
- Primary tool returned off-target results AND fallback available
- Only tried 1 tool and results seem incomplete
Underperformance Detection:
After each tool call, LLM inspects results:
python# Pseudo-logic for LLM reasoning:

        # Example: User wanted fiction, got non-fiction
        # Example: User wanted fantasy, got romance

    elif len(tool_result.books) < 30:
        decision = "try_additional"
        reasoning = "Insufficient candidates"
    else:
        decision = "stop"
        reasoning = "Good results, sufficient quantity"
Relevance Assessment Logic:

LLM examines book subjects, genres, tones in tool results
Compares against query keywords and intent
Classifications:

"good match": Majority of books align with query
"off-target": Majority of books don't match (wrong genre/subject)
"mixed": Some matches, some misses
"insufficient": Too few books to assess

Phase 5: Orchestrator Updates
Goal: Wire together PlannerAgent → CandidateGeneratorAgent → CurationAgent
5.1 New Orchestrator Flow
pythondef execute(request: AgentRequest) -> AgentResponse:
    # Stage 1: Planning
    planner_result = self.planner.execute(request)
    strategy = planner_result.strategy
    profile_data = planner_result.profile_data
    
    # Stage 2: Candidate Generation
    generator_input = {
        "query": request.user_text,
        "strategy": strategy,
        "profile_data": profile_data
    }
    generator_result = self.candidate_generator.execute(generator_input)
    candidates = generator_result.candidates
    tool_executions = generator_result.tool_executions
    
    # Stage 3: Curation
    execution_context = ExecutionContext(
        planner_reasoning=strategy.reasoning,
        tool_executions=tool_executions,
        profile_data=profile_data
    )
    curation_result = self.curation_agent.execute(
        request=request,
        candidates=candidates,
        execution_context=execution_context
    )
    
    return curation_result
5.2 Fallback Handling
Planner Failure:

Use hardcoded fallback strategy
Warm user → als_recs, cold user → popular_books

Generator Failure (no candidates):

Return helpful error message
Check if query mentions recent years (catalog is ≤2004)

Curator Failure:

Return first 10 candidates with generic prose
Partial success response

5.3 State Transfer
No shared mutable state - orchestrator manages data flow:

Planner produces strategy → passed to Generator
Generator produces candidates → passed to Curator
Each agent is stateless (functional)
