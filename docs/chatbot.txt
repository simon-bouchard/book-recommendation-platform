PROJECT STATE (NOW)

Stack: FastAPI + Jinja templates + static JS.

Notable routes/files:

routes/chat.py → /chat page + POST /chat/agent (returns {reply, books[]}).

app/agents/web_agent.py → ReAct-style agent with external tools (DuckDuckGo, Wikipedia, OpenLibrary, local help-docs).

models/… → core ML engines: ALS (warm), ColdHybrid (subject+Bayesian), similarity strategies (subject, ALS, hybrid), attention pooling for subjects, ModelStore singleton.

Current demo:
The agent is a virtual librarian. It defaults to no tools for greetings/small talk. For site-help questions it uses local help-docs. For external facts it may call web tools (DuckDuckGo, Wikipedia, OpenLibrary). Responses normalize to a single line starting with Final Answer:; /chat/agent strips that prefix.

Constraints in place:
3-tool cap, deduped queries, temperature=0, never invent titles.

CONVERSATION STATE & REDIS (NOW)

On first request, server sets a conv_id cookie (SameSite=Lax).

Each turn is stored in Redis under chat:{conv_id} as a rolling JSON list of last N turns (default 3).

Before calling the agent, the server prepends:

(User profile context — if enabled; favorite subjects + recent interactions from SQL).

(Conversation so far — last few turns).

The new user message.

This lets the agent resolve follow-ups naturally (“like the second one you mentioned…”).

TTL: 2 days.
Failure mode: if Redis is down, degrade to stateless.
Security: Redis local only; no DB writes from the agent.

WHAT I AM BUILDING NEXT (ML-INTEGRATED AGENT)

The chatbot will become a grounded, catalog-locked RAG-style agent. Instead of inventing, it will call the same ML models already used by the website. External web tools remain, but will only be used for clearly external queries. All recommendation answers will come strictly from internal models.

Internal tools (atomic, deterministic):

User-centric recs

als_recs(user_id, top_k) → ALS baseline (warm).

cold_hybrid_recs(user_id|fav_subjects_idxs, top_k, w, tiers) → Cold hybrid subject+Bayesian.

Item-centric sim

subject_sim(item_idx, top_k) → subject embeddings.

als_sim(item_idx, top_k) → ALS similarity (if available).

hybrid_sim(item_idx, top_k, alpha, min_count) → blend (if both available).

Meta join

fetch_book_meta(item_idxs[]) → authoritative {title, author, year, cover_url}.

Help/docs

help-list, help-read, help-search remain for onboarding and site explanations.

Tool availability is gated:

ALS tools are only exposed if the user/book exists in ALS.

Hybrid sim only if both ALS + subject are present.

Cold tools always available.
Unavailable tools are simply not listed for the agent.

AGENT BEHAVIOR

Persona: concise librarian; never invents; always catalog-grounded.

Tool policy:

No tools for greetings/small talk.

For recs, prefer internal tools.

Web tools only if the user clearly asks for external context.

Chaining: Agent may call multiple tools in one turn (e.g., ALS recs → then subject sim for diversity → merge/dedupe → final answer).

Finalization: Exactly one line starting with Final Answer: plus short explanation with provenance labels (e.g., “(source: ALS baseline)”).

Memory: Rolling Redis window and optional profile context guide tool choice; outputs always grounded in the current turn’s fetched IDs.

NON-GOALS

No DB writes.

No scraping.

No second agent or hard “mode” toggle—one agent dynamically chooses between internal and web tools.

ACCEPTANCE CRITERIA

/chat loads; /chat/agent works.

Small-talk → short reply, no tools.

Internal rec queries → only catalog results, correct provenance.

Site-help queries → answered via local docs.

Web queries → use web tools, never for recs.

Redis history allows follow-up references.

All outputs end with one Final Answer: line and never invent titles.
