CONVERSATION STATE & REDIS (NOW)

Goal: make the chatbot aware of the ongoing conversation instead of just the last message.

How it works (server-side, no login required):

On first request, the server sets a conv_id cookie (SameSite=Lax).

Each turn is stored in Redis under chat:{conv_id} as a tiny JSON list of the last N exchanges (default HIST_TURNS=3; i.e., up to 3 {user, assistant} pairs).

Before calling the agent, the server prepends a compact context block built from that rolling history:

(Conversation so far — use only for context)
User: …
Assistant: …
…


followed by the new user message. This lets the agent refer back naturally (e.g., “the last book you mentioned…”).

TTL & limits: the Redis key has a TTL (default CHAT_TTL_SEC=172800, i.e., 2 days). Only a short window is kept to control token usage.

Environment variables:

REDIS_URL (e.g., redis://localhost:6379/0)

CHAT_TTL_SEC (default 172800)

CHAT_HIST_TURNS (default 3)

Failure mode: if Redis is unavailable, the system degrades to stateless (no history) but still responds normally.

Security/ops: default Redis binding is local; memory is ephemeral and scoped to the browser session via cookie. No user login or DB writes involved.
