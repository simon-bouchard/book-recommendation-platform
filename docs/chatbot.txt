Multi-Agent System Plan (Expanded)
Keep (re-use unchanged components)

HTTP boundary:
routes/chat.py remains the single FastAPI entrypoint. It continues to handle rate-limits, cookies, history persistence, and response formatting. The outward API contract (ChatOut(reply, books[])) does not change.

It still calls normalize_visible_reply, extract_book_ids_from_steps, and build_books_from_ids to produce book cards.

It still logs request_start and request_end events with conv_id and uid.

Context building:
context_builder.py + user_context.py for composing user profile context and recent interaction snippets. Only the Recsys branch will use this. Docs and Web branches receive raw user text only.

Settings:
settings.py continues to centralize LLM/provider toggles, embedding factory, Redis URL, rate limits. Extend it later with new Conductor/Router knobs (timeouts, budgets, critic toggle).

Tooling surface:
ToolRegistry + existing tool modules (web.py, help.py, internal_tools.py, semantic_search.py, subject_search.py). These remain the authoritative adapters for external and internal capabilities.

Prompt rules:
The strict AGENT_PROMPT stays as the backbone of the Recsys agent. It enforces the “internal tool → return_book_ids → Final Answer” ritual, the subject/ALS/semantic chooser, and the retry rule for subject pools.

Change (split responsibilities)

Replace the single web_agent.py with a Conductor + Router + three branch agents.

The Router is very light: it inspects the raw user text (and possibly a warm/cold signal via user_num_ratings) and returns "recsys" | "web" | "docs".

The Conductor calls Router, dispatches to the appropriate branch, and handles any optional Web-for-enrichment merge.

Profile context (build_composed_input) is not used in Router. It is only built and consumed inside the Recsys branch.

New files

app/agents/conductor.py — Orchestration entrypoint. Calls Router, dispatches to the chosen branch agent, optionally merges Web enrichment into Recsys answers, returns a unified ConductorResponse.

app/agents/router.py — Deterministic branch classifier. Very simple rules mapping user text patterns to "recsys", "web", or "docs".

app/agents/contracts.py — Typed result envelopes (RecsysResult, WebResult, DocsResult, ConductorResponse) for testing/log stability.

app/agents/budget.py — Per-request budget ledger (max tool calls, max elapsed time). Conductor checks before dispatching tools.

app/agents/tools_adapters.py — Async wrappers around existing tool factories, with timeouts and span logging. Conductor calls these, not LangChain primitives.

app/telemetry/tracing.py — OpenTelemetry integration. One span per conductor, router, branch, and tool call. Attach conv_id and budget counters.

(later) app/agents/critic.py — Optional validator for provenance (e.g. Recsys outputs must include IDs, Web outputs must include citations, Docs outputs must come from help-read).

Branch agents
Recsys branch

Input: full composed input (profile + history + raw user text).

Tools: ALS (als_recs), Subject hybrid (subject_hybrid_pool + subject_id_search), Semantic search (book_semantic_search, seeding only), and return_book_ids.

Rules:

Must call at least one internal tool, then finalize with return_book_ids.

Semantic search only for seeding or identification; never a finalizer.

Retry subject_hybrid_pool once with higher w if initial pool is too popularity-heavy.

LLM curates from a bounded shortlist (union by item_idx, capped per tool) and chooses 4–12 before return_book_ids. No numeric fusion outside the tools.

Web branch

Input: raw user text only.

Tools: web-search, Wikipedia, OpenLibrary.

Rules:

For facts, “what is / who wrote / latest / compare / links”.

Never outputs item_idx.

May run in parallel with Recsys for enrichment (short blurbs, publication info), but only Recsys supplies IDs.

Docs branch

Input: raw user text only.

Tools: help-manifest, help-read.

Rules:

For onboarding and “how does this site work” questions.

Deterministic path: manifest → choose alias → help-read.

No Web or internal tools enabled here.

Router agent

Input: raw user text (+ optional user_num_ratings warm/cold signal).

Rules:

“profile, ratings, chatbot, portfolio, how does X work” → Docs.

“what is, who wrote, compare, best/top/latest, fresh info” → Web.

“recommend, subject, tone, vibe, forgot the title” → Recsys.

Output: compact plan object, e.g. {"target":"recsys","intent":"recommend"}.

Candidate fusion (inside Recsys only)

Do not interleave blindly. Select tools based on intent (ALS for warm, Subject for subject cues, Semantic for free-text seeding).

Each tool returns a pool of book dicts (item_idx, title, author, etc.).

Build a union by item_idx, collapse overlaps, and cap the total count (intent-aware caps).

For each candidate, create a micro-card: {item_idx | title — author | src:[ALS/SUBJ/SEM] | hints}.

Present these compact cards to the LLM for curation.

LLM applies the prompt rules (curation, diversity, retry if needed) and must call return_book_ids.

Observability

Keep capture_agent_console_and_httpx to pipe raw LangChain + httpx traces into logs/chatbot.log.

Add structured spans for conductor, router, each branch, and each tool call.

Attach conv_id, uid, plan, and budget counters as attributes in both logs and traces.

Config

Extend ChatSettings with:

enable_critic, app_max_tools, internal_timeout_ms, web_timeout_ms, router_debug_mode, app_latency_ms.

Migration path

PR-1 (scaffolding, no user-visible change)

Add Conductor, Router, Contracts, Budget, Adapters, Telemetry stubs.

Switch /chat/agent from web_agent.answer to Conductor.run, but delegate everything to the current single agent (as Recsys).

Keep output identical to today.

PR-2 (true MAS)

Enable Web and Docs branches with gated tool registries.

Router chooses branch.

Recsys agent enforces internal tool + return_book_ids.

Web returns snippets + citations.

Docs returns onboarding answers.

Add provenance and branch timings to logs.

Future evolution

Split Recsys into multiple sub-agents (ALS-specialist, Subject-specialist, Semantic-specialist) coordinated by a local planner.

Add Critic to validate provenance and polish tone.

Extend Router to handle hybrid Web+Recsys enrichment plans explicitly.
