# Recommendation Agent Evaluation

Comprehensive testing suite for the two-stage recommendation agent (Retrieval → Curation).

## Overview

This evaluation suite tests the recommendation agent with:
- **Part 1: Tool Selection Tests** - Deterministic checks that agent calls appropriate tools
- **Part 2: Two-Stage Integration Tests** - Validation of retrieval → curation pipeline

## Testing Strategy

**Critical Design Decision:** The orchestrator returns only the curation agent's response, which doesn't include the retrieval agent's tool executions. Therefore:

**For Tool Selection Tests** (`tool_selection_*`):
- Tests **retrieval agent directly** (not full orchestrator)
- Allows inspection of tool executions
- Validates: tool selection, candidate gathering, profile ordering
- **Output checks**: 
  - Candidates: ≥20 (can be 80-300, that's normal)
  - No prose (retrieval doesn't generate prose)
  - No final book count check (curation not tested here)

**For Integration Tests** (`two_stage_integration`):
- Tests **full orchestrator** (retrieval → curation pipeline)
- Validates end-to-end workflow
- **Output checks**:
  - Final books: 3-30 (after curation selects from candidates)
  - Prose: Required (generated by curation)
  - Can't inspect individual tool calls (abstracted by orchestrator)

## Test Structure

### Test Categories

1. **tool_selection_warm_user** (6 tests)
   - Tests for users with ≥10 ratings (warm users)
   - Covers vague, descriptive, and genre queries
   - Validates ALS recommendations usage

2. **tool_selection_cold_user** (5 tests)
   - Tests for users with <10 ratings (cold users)
   - Should use subject_hybrid_pool instead of ALS
   - Validates fallback behavior

3. **two_stage_integration** (4 tests)
   - Tests retrieval → curation handoff
   - Validates metadata flow between stages
   - Tests orchestrator fallback logic

4. **edge_cases** (2 tests)
   - Empty queries
   - Very long queries
   - Error handling

## Query Types

### Vague Queries
- "recommend something good"
- "what should I read"

**Expected behavior:**
- Should call `user_profile` tool FIRST (if profile access granted)
- Then use ANY appropriate retrieval tool based on profile insights:
  - Could use `als_recs` (personalized recommendations)
  - Could use `book_semantic_search` (if profile suggests specific vibes)
  - Could use `subject_hybrid_pool` (if profile shows clear subject preferences)
- Key rule: **profile called before retrieval**, not which specific tool

### Descriptive Queries
- "dark atmospheric mystery with gothic vibes and psychological tension"
- "cozy heartwarming story with small town charm and feel-good moments"

**Expected behavior:**
- Should use `book_semantic_search` (works best with 4+ word descriptions)
- Should NOT call `user_profile` (query is already specific)

### Genre/Subject Queries
- "historical fiction"
- "science fiction with space opera and military themes"

**Expected behavior:**
- Should use `subject_id_search` to resolve subject names
- Then use `subject_hybrid_pool` to get books
- Can combine up to 3 subjects

## What Gets Tested

### Layer 1: Tool Selection (Deterministic)
No LLM-as-judge needed - just inspect tool execution list:

✓ Correct tool selection for query type
✓ user_profile called before retrieval (for vague queries)
✓ Warm users use als_recs
✓ Cold users use subject_hybrid_pool
✓ Descriptive queries use semantic_search
✓ Genre queries use subject tools

### Layer 2: Output Structure

**For Tool Selection Tests (Retrieval Only):**
✓ Minimum candidate count (≥20, can be 80-300)
✓ No prose expected (retrieval doesn't generate)
✓ No final book count check (curation not tested)

**For Integration Tests (Full Pipeline):**
✓ Final book count (3-30 books after curation)
✓ Response has prose (curation generates explanation)
✓ Books have proper metadata

### Layer 3: Integration
✓ Retrieval stage gathers candidates
✓ Retrieval creates summary for curation
✓ Curation receives candidates and context
✓ Final response combines books + prose

## Running Tests

Requires `DATABASE_URL` environment variable to be set (same as main application).

```bash
cd evaluation/chatbot/recommendation_agent
python evaluate_recommendation.py
```

The tests will:
1. Connect to your actual database
2. Find users with appropriate rating counts for each test
3. Execute agents with real tool calls
4. Validate tool selection and results

## Expected Pass Rates

- **Tool Selection**: 85-95% (deterministic checks)
- **Integration**: 80-90% (depends on data quality)
- **Overall**: 80-90% initially

## Database Requirements

Tests need access to your production/staging database with:
- Users table with ratings/interactions
- Books table with metadata
- Subjects table for genre searches
- Semantic search index (for semantic_search tool)

The evaluation script will automatically find users with appropriate rating counts for each test scenario.

## Test Case Format

```json
{
  "name": "vague_query_warm_with_profile",
  "query": "recommend something good for me",
  "user_id": 278859,
  "user_state": {
    "num_ratings": 12,
    "allow_profile": true,
    "is_warm": true
  },
  "expected_tools": {
    "should_use_user_profile": true,
    "profile_before_retrieval": true,
    "should_use_any_retrieval_tool": true
  },
  "expected_output": {
    "min_candidates": 20,
    "min_final_books": 3,
    "max_final_books": 30
  }
}
```

## Adding New Tests

1. Add test case to `test_cases.json`
2. Specify `user_id` (must exist in database)
3. Specify user_state (num_ratings, allow_profile)
4. Define expected_tools based on query type
5. Set expected_output constraints
6. Run evaluation to validate

Note: Use existing test users (278859, 278857, 278867) or add new test users to database first.

## Key Design Decisions

✅ **Use real tools, no mocking** - Tools are fast and free, gives more confidence

✅ **Use real database** - Tests execute against actual data for realistic validation

✅ **Deterministic tool checks** - No LLM-as-judge for tool selection (faster, cheaper)

✅ **Focus on structure** - Don't test factual accuracy (separate enrichment quality tests)

✅ **Test both warm and cold paths** - Different tool availability changes behavior
